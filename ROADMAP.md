# 🗺️ v3 / v4 Roadmap — 雲端同步與 AI 資料自動化

### 核心定義：極簡化的「整理與紀錄」
經過討論，v3 將捨棄不必要的複雜狀態（不分欄、不設年度重置），一切回歸最直覺的操作：**「紀錄我買了哪些零股，並能跨裝置同步與匯出。」** 
而針對資料來源不透明的痛點，在 v4 導入 LLM 自動化分析，大幅降低我們手動維護的成本。

---

## 🏗️ 架構設計決策

在進入實作之前，我們確認了以下關鍵的技術方向：

### 資料架構：全部集中至 Supabase
| 資料類型 | Supabase 資料表 | 誰在讀 | 誰在寫 |
|---|---|---|---|
| 公開股票資料（~900 筆） | `stocks`（公開讀取） | 所有訪客 | GitHub Actions（一週三次） |
| 使用者個人持有清單 | `user_stocks`（RLS 保護） | 僅本人 | 使用者前端（批次同步） |

所有資料統一存放在 Supabase PostgreSQL，前端透過 Supabase JS SDK 存取。

### GitHub Actions 自動更新公開資料
- **頻率**：每週一、三、五，台灣時間 18:00（UTC 10:00，收盤後）
- **流程**：執行 Python 腳本 → 從證交所/櫃買中心抓最新股價 → **比對 Supabase 現有資料，只更新有變動的部分** → 無變動則跳過
- **好處**：前端不需要下載整包靜態檔，直接從 DB 查詢即可；資料更新完全自動化

### 使用者持有清單的批次同步策略（Batch Sync）
為減少不必要的 API 請求，使用者的 🛒 買入/取消操作**不會即時打 API**，而是：
1. **點擊 🛒** → 只更新 `localStorage`（零延遲，與目前體驗完全一致）
2. **自動同步**：每 15 分鐘自動偵測是否有未同步的變更，有的話批次上傳
3. **手動同步**：控制列新增 `[☁️ 儲存至雲端 (N)]` 按鈕，`N` 顯示未同步的筆數
4. **離線容錯**：斷網時所有操作仍存在 localStorage，恢復網路後一次同步

## 🚀 v3：雲端持續化與匯出管理 (Current Focus)

**技術選型：Supabase**（Auth + PostgreSQL + Row Level Security）

### Phase 1：Supabase 雲端會員系統
- **註冊與登入**：Google OAuth / Email Magic Link。
- **UI 更新**：Header 區塊新增登入按鈕與使用者狀態圖示。
- **模組化**：新增 `auth.js` 處理 Supabase token 與 session。
- **資料庫設計 (PostgreSQL)**：只紀錄「誰」買了「哪一檔」，以及可選的「個人備註」。
  ```sql
  CREATE TABLE user_stocks (
    user_id     uuid REFERENCES auth.users NOT NULL,
    stock_id    text NOT NULL,
    note        text,
    created_at  timestamptz DEFAULT now(),
    PRIMARY KEY (user_id, stock_id)
  );
  ALTER TABLE user_stocks ENABLE ROW LEVEL SECURITY;
  ```

### Phase 2：無縫儲存與同步 (`sync.js`)
- **已登入**：透過批次同步策略，將 localStorage 的變更定期上傳至 Supabase `user_stocks`。
- **未登入（遊客）**：維持現狀存在 `localStorage`，確保新訪客開箱即用。
- **首次綁定**：當遊客首次登入時，一鍵將 `localStorage` 已買入的紀錄上傳合併至雲端帳號。

### Phase 3：匯出代領清單
- **一鍵匯出 CSV**：於控制列新增 `[📥 匯出清單]` 按鈕，直接下載針對代領業者友善格式的 `.csv` 檔案（股號、公司名、個人備註）。

## 🤖 v4：MOPS 爬蟲與 LLM 資料自動化擷取 (Future Plan)

為了解決「找不到紀念品資訊」與「人工維護成本過高」的問題，v4 將後端 Python 腳本全面升級，實作自動化資料獲取流程。

### 為什麼不爬 `股東e票通`？
- e票通需要憑證登入（自然人/券商憑證），無法寫自動爬蟲。
- e票通只包含議案，**不包含**實體紀念品發放資訊。

### v4 解決方案：MOPS + LLM 解析
台灣證券交易所的「公開資訊觀測站 (MOPS)」是法定公告大廳，但其「紀念品發放原則」欄位是由各家股務手打的「非結構化中文文字」。

**實作流程：**
1. **溫柔爬蟲 (Polite Scraper)**：
   撰寫 Python 爬蟲，每天固定排程（如透過 GitHub Actions）去 MOPS 抓取當日發布的「召開股東常會公告」。
2. **LLM 非結構化資訊萃取 (Data Extraction)**：
   將爬取到的中文發放原則送入 GPT / Gemini API。
   ```text
   Prompt Example: 
   "請分析以下文字：『除親自出席股東會者外，本公司將不予發放紀念品...』
   回傳 JSON: { 
     'has_souvenir': true, 
     'item': '無', 
     'value': 0, 
     'zero_shares_eligible': false 
   }"
   ```
3. **自動化更新**：
   LLM 解析回來的結構化 JSON，自動更新專案的 `StockData.xlsx` / `data.json`，讓前端永遠保持最新鮮的「高 CP 零股清單」，零人工維護成本！

### 新增資料表：`voting_status`（投票現況）

除了歷史 CP 值評估表之外，v4 將新增一張「投票現況」表，持續抓取並更新**目前正在開放投票且會發紀念品**的股票清單。

**資料來源**：[histock.tw/stock/gift.aspx](https://histock.tw/stock/gift.aspx)

| 欄位 | 說明 |
|---|---|
| `stock_id` | 代號 |
| `name` | 名稱 |
| `price` | 目前股價（引用 `stocks` 主表的最新股價） |
| `last_buy_date` | 最後買進日（停止過戶日前 2 個營業日） |
| `meeting_date` | 股東會日期 |
| `meeting_type` | 性質（常會 / 臨時會） |
| `meeting_location` | 開會地點 |
| `souvenir` | 股東會紀念品 |
| `zero_share_mail` | 零股寄單（是否寄發開會通知給零股股東） |
| `cp_ratio` | 性價比（紀念品價值 ÷ 股價） |

**更新機制**：
- 與公開資料相同頻率：GitHub Actions 每週一、三、五台灣時間 18:00 爬取 histock 更新
- 過期的股東會資料自動標記為已結束
- 前端可新增一個「📋 投票現況」的專區或篩選器，讓使用者一眼掌握該上車的時機

### 資料回流：股東會結束後更新主表 `stocks`

當股東會結束後，`voting_status` 裡確認的紀念品資訊將自動回流至主表 `stocks`，更新以下欄位：

| `stocks` 欄位 | 更新來源 |
|---|---|
| `stock_id` | 代號 |
| `name` | 公司名稱 |
| `price` | 最新股價（由 Actions 定時更新） |
| `last_souvenir` | 上次紀念品 ← 從 `voting_status.souvenir` 寫入 |
| `history_5y` | 五年內發放 ← 累加紀錄（上限 5 筆） |
| `cp_score` | 性價比 ← 維持現有公式，不改動 |
| `last_condition` | 去年條件 ← 從 `voting_status` 的零股寄單等資料帶入 |
| `star_rating` | 推薦評分 ← **新公式：多維度加權**（見下方） |

#### 歷史紀錄管理：滾動 5 年窗口

每次新資料回流時，**淘汰超過 5 個年度前的紀錄**（`year >= 當前年 - 4`），不限筆數。
- 某公司 5 年只發 3 次 → 保留 3 筆，CP 公式的 `(次數/5)` 自動折減
- 某公司 5 年發了 8 次（如每年常會 + 臨時會）→ 保留 8 筆，CP 值反映實際多次發放的好處
- 不需要年底清理腳本，回流時順便淘汰即可

#### 推薦評分公式（星星）

星星不再只是 CP 值的分級，而是一個**綜合推薦度**：

**v3 暫行公式**（現有資料不包含零股領取條件）：
```
推薦分數 = (CP 值分數 × 0.6) + (穩定度 × 0.4)
```

| 維度 | 衡量 | 計算方式 | 滿分 |
|---|---|---|---|
| CP 值分數 | 花多少錢拿多少東西 | 現有 CP 值正規化至 0~100 | 100 |
| 穩定度 | 會不會明年就不發了 | `連續發放年數 ÷ 5 × 100` | 100 |

**v4 完整公式**（待 MOPS/histock 爬蟲 + LLM 解析上線後）：
```
推薦分數 = (CP 值分數 × 0.4) + (穩定度 × 0.3) + (零股友善度 × 0.3)
```
零股友善度：可電投領 = 100、要親自出席 = 30、不給領(整張限定) = 0

星星對應：⭐5 = 80+、⭐4 = 60~79、⭐3 = 40~59、⭐2 = 20~39、⭐1 = <20（門檻值可微調）

**完整資料生命週期**：
```
histock 爬蟲 → 寫入 voting_status → 前端顯示「投票現況」
                                      ↓ 股東會結束
                            回流寫入 stocks 主表（保留最新 5 筆）
                                      ↓
                            重算 CP 值 + 推薦評分（三維度）
                                      ↓
                            前端主表格自動反映最新紀念品資訊
```

### 前端讀取畫面（Loading Screen）

由於 Supabase 免費方案在閒置後有冷啟動延遲（5-10 秒），前端需加入讀取畫面以優化體驗：
- 網頁開啟時先顯示品牌 Logo + 骨架畫面（Skeleton UI）或載入動畫
- 背景同時發起 Supabase 資料請求
- 資料回傳後平滑過渡至完整頁面，避免空白尷尬
