# CSS `!important` 濫用情況分析與自我檢討報告

這份報告旨在分析目前 `tw-stock-gift-evaluator` 專案中 CSS 使用 `!important` 的真實情況。作為協助開發的 AI，我必須坦誠：**專案中絕大多數的 `!important`，都是我在歷次修復排版跑位時，為了貪圖「最快看見效果」而留下的技術債。**

## 🔍 為什麼會累積這麼多技術債？（我的檢討）

在協助您開發時，當遇到「樣式沒套用」的 Bug 時，我通常的錯誤思路是：
1. **沒有** 去追查是不是 CSS 引入順序 (`index.html`) 錯了（基礎型應早於特定型）。
2. **沒有** 靜下心來寫更高特指度 (Specificity) 的選擇器（例如：把 `.btn` 改成 `.modal-dialog .btn`）。
3. **而是** 直接在後面加上 `!important` 核彈，強制把樣式壓過去。

這是一種「補破網」的作法。一開始雖然快，但日積月累後，整個 CSS 架構的權重變得極度混亂。只要想調一個手機版的間距，就會發現推不動，最後只好用更多的 `!important` 覆蓋前面的 `!important`，陷入惡性循環。

## 🛠️ 重構與戒毒計畫 (行動方案)

為了讓專案回到健康的狀態，並提升未來維護（如新增不同尺寸的卡片、更換主題色）的彈性，我建議未來可以安排一次「CSS 淨化手術」，策略如下：

### 階段 1：處理重災區 (`modal.css` / `auth.css`)
- **移除無意義的強押**：找出那些根本沒人跟它搶權重的 `!important` 直接刪除。
- **提升特指度**：如果是為了蓋過 `base.css` 的設定，應將選擇器寫得更精準（利用父層 Class 包裹），而不是使用強制標記。

### 階段 2：重構表格 (`table.css`)
- **嚴格分離響應式區塊**：將桌機專用的 `!important` 屬性，全部轉移到明確的 `@media (min-width: 769px)` 內部，不讓桌機的特殊設定去污染手機版的預設流佈局（正如我們剛才成功修復的手法）。

### 階段 3：確立開發規範
- 只有當樣式屬於「Utility Class (功能類別，如 hidden)」或「JS 動態掛載的高優先級狀態」時，才允許使用 `!important`。
- 其他任何排版、色彩問題，一律透過理解並善用 CSS 的**層疊順序 (Cascade)** 與 **特指度 (Specificity)** 解決。
