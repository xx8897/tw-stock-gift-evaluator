# 知識樹深度解析

> 樹的最後一個子節點叫「雜物」，base_value 是 0，keywords 有 37 個。這不是偷懶，是對長尾問題的誠實回答。

## 樹的頂層結構

```
🎁 股東會紀念品大分類樹 (base_value: 40, receiving_cost: 20)
├── 💰 禮物卡 (面額直接捕捉)
├── 🎫 票券與高價特賞 (value_multiplier: 0.7)
├── ⚡ 3C 與家用電器
├── 🍳 寢具與大型烹飪器具
├── 🛠️ 五金工具與戶外用品
├── 🍽️ 不鏽鋼/收納/餐用具
├── 🍔 食品生鮮與日用雜貨
├── 💊 保健品專區 (value_multiplier: 0.5)
└── 雜物 (base_value: 0)
```

九大類別加上一個「雜物」桶。每個類別下面還有子類別，子類別下面還有具體項目，最深可以到四層。

## 匹配邏輯的優先順序

DFS 搜尋時，每個節點的匹配有嚴格的優先順序：

```
1. keywords_not（排除優先）
   → 任一關鍵詞出現 → 不匹配，停止往下搜

2. keywords（OR 匹配）
   → 任一關鍵詞出現 → 匹配，繼續往深搜

3. keywords_and（AND 匹配）
   → 所有關鍵詞都出現 → 匹配，繼續往深搜

4. regex（正則匹配）
   → 正則匹配 → 匹配，繼續往深搜
   → 可以捕捉群組（如面額數字）

5. pass_through（穿透）
   → 不匹配但允許繼續往深搜
```

**排除為什麼優先？** 因為「卡套」包含了「卡」，「馬克杯」包含了「杯」，「撲克牌」包含了「克」和「牌」。如果不先排除，這些東西都會被誤分到「禮物卡」或「餐具」類別。

## 繼承：子節點可以不設值

「星巴克隨行杯」的節點：

```json
{
  "id": "starbucks_tumbler",
  "name": "星巴克隨行杯",
  "type": "item",
  "rules": {
    "keywords": ["星巴克", "隨行杯"],
    "keywords_not": [],
    "keywords_and": [],
    "regex": ""
  },
  "attributes": {
    // 注意：沒有 base_value！
  }
}
```

沒有 `base_value` 怎麼辦？`_resolve_attr()` 會向上找：

```
星巴克隨行杯 → 沒有 base_value
  ↑
☕ 飲品/咖啡 → base_value: 80
```

找到 80。這就是繼承。

**`value_multiplier` 也會繼承。** 「保健品專區」設了 `0.5`，所有子類別——維他命、魚油、益生菌——估值自動減半。一個改動，傳播到所有子節點。

## 標籤穿透：決定代領成本

當「王品」匹配到「票券與高價特賞」類別時，`_collect_tags()` 會收集路徑上所有祖先的標籤：

```
王品
  ↑ 票券與高價特賞 → tags: ["票券"]
  ↑ root → tags: []
```

結果：`["票券"]`。代領成本 = 15 元（票券類通常不需親自到場，代領成本較低）。

如果匹配到「不鏽鋼保溫杯」：

```
不鏽鋼保溫杯
  ↑ 保溫杯 → tags: []
  ↑ 餐用具 → tags: []
  ↑ 不鏽鋼/收納 → tags: []
  ↑ root → tags: []
```

結果：`[]`。代領成本 = 20 元（實物通常需要親自領取或委託他人，成本較高）。

## 正則捕捉：面額直接從名字裡提取

「面額直接捕捉區」是知識樹裡最巧妙的設計之一：

```json
{
  "id": "面額捕捉_全聯",
  "name": "全聯禮物卡面額捕捉",
  "type": "item",
  "rules": {
    "keywords_and": ["全聯"],
    "keywords_not": ["購物袋", "環保袋"],
    "regex": "(\\d{1,3}(?:,\\d{3})*)元"
  },
  "attributes": {
    "base_value": 0
  }
}
```

當 `base_value` 是 0 時，表示「用正則捕捉到的數字作為估值」：

```
輸入：「全聯禮物卡500元」
regex 捕捉："500"
base_value = 0 → 使用捕捉值 500
value_multiplier = 1.0（禮物卡面額即估值）
receiving_cost = 15（票券標籤）
最終估值 = max(0, int(500 × 1.0) - 15) = 485
```

這一個正則表達式取代了 V4 裡幾十行 if-else——不需要為「100元禮物卡」、「200元禮物卡」、「500元禮物卡」各寫一個判斷。

## 雜物桶：對長尾的誠實回答

```json
{
  "id": "misc_catch_all",
  "name": "雜物",
  "type": "item",
  "rules": {
    "keywords": ["油票", "面紙", "衛生紙", "毛巾", "香皂", "洗手乳", "口罩",
                 "酒精", "乾洗手", "濕巾", "水壺", "便當盒", "環保杯",
                 "功能", "吊衣架", "掛鉤", ...總共 37 個],
    "keywords_and": [],
    "keywords_not": [],
    "regex": ""
  },
  "attributes": {
    "base_value": 0
  }
}
```

`base_value: 0` 意味著落在雜物桶的紀念品估值為 0——不是不值錢，是我們不知道它值多少。最終估值 = `max(0, int(0 × 1) - 20) = 0`。

這是對長尾問題的誠實回答：有些紀念品就是很難估價，不如誠實地給 0，讓星星系統把它們降到 0 星，而不是給一個不准確的估值。

## 複合禮物：拆分後只扣一次代領成本

```
輸入：「不鏽鋼碗+保溫杯500ml&衛生紙」
  │
  ├── 拆分（、&、與、和、及）
  │   ├── "不鏽鋼碗" → DFS → "碗/盤/碟" → base_value: 80
  │   ├── "保溫杯500ml" → DFS → "保溫杯" → base_value: 120
  │   └── "衛生紙" → DFS → "雜物" → base_value: 0
  │
  └── 合計：(80 + 120 + 0) × 1.0 - 20 = 180
      代領成本只扣一次（最上層呼叫）
```

五個分隔符：`+`、`&`、`與`、`和`、`及`。這是台灣股東會紀念品最常見的複合形式。

## 年份剥离：防止數字干擾

在 DFS 搜尋之前，`estimate_5year_total()` 會先用正則剝離年份：

```python
# 剝離年份前綴，防止 "2025" 被匹配到 "20" 元禮物卡
item_name = re.sub(r'^\(?\d{4}\)?', '', item_name.strip())
```

這個小動作防止了一個微妙的 bug：如果五年紀錄格式是 `(2024) 200元禮物卡`，不剝離的話「2024」裡的「20」可能匹配到「20」元禮物卡節點。

## 模擬器：測試你的規則

`gift_tree/app.py` 提供了一個 Flask 端點：

```
POST /api/simulate
```

把所有 1,300+ 個實際紀念品跑過分類樹，返回：

- 已分類的項目（落在哪個節點、估值多少）

視覺編輯器的前端 JS 也有一個 `classifyItemJS()` 函數——DFS 演算法的 JavaScript 重新實作。這讓編輯器可以在前端即時預覽分類結果，不需要呼叫後端。一致的行為，不同的語言，兩次實作。
