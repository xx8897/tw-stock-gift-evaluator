# 真實教訓

> 這些教訓不是從書上來的。是從一個真的有人用的產品中長出來的。

## 技術決策

### 教訓一：把會變的東西從代碼移到資料

V4 的 if-else 鏈裡包含了所有商品知識。每次有新的紀念品類型出現，就要改代碼、測試、部署。V5 把所有知識搬到了 JSON 樹裡，代碼只做搜尋和計算。

這不是新思想——規則引擎、決策表、配置驅動開發都在說同樣的事。但 V5 把它做到極致：1,680 行的商品知識，200 行的通用演算法，零行硬編碼估價邏輯。

判斷什麼該從代碼移到資料的方法：如果一個非工程師（但懂業務的人）可以修改它，它就應該是資料。V5 的視覺編輯器讓這個判斷變成了現實。

### 教訓二：DFS 搜尋比 if-else 更容易維護

if-else 鏈的排序敏感、排除條件散落、每次新類別要插入正確位置。DFS 搜尋的匹配邏輯統一（NOT → OR → AND → REGEX → pass_through）、節點獨立、新增類別只需要加一個子節點。

最重要的差異：DFS 搜尋是聲明式的（「我想要什麼」），if-else 鏈是命令式的（「先做這個，再做那個」）。聲明式更容易理解、更容易修改、更容易測試。

### 教訓三：零後端不等於沒有架構

GitHub Pages + Supabase + GitHub Actions。每月成本接近零。但架構的複雜度沒有消失，只是轉移了：

- Supabase 處理資料庫和認證——你不需要管伺服器，但你需要管 Schema、VIEW、RLS
- GitHub Actions 處理定時更新——你不需要管排程器，但你需要管 API 限流、錯誤處理
- GitHub Pages 處理靜態託管——你不需要管 CDN，但你需要管快取、載入時間

零後端的代價是：你用別人的基礎設施，遵守別人的限制。免費方案的 Supabase 有連線池限制、GitHub Actions 有 6 分鐘超時、GitHub Pages 不支援伺服器端渲染。

## 產品設計

### 教訓四：閘門邏輯比加權平均更誠實

CP+ 5.0 但頻率 1/5 的股票，加權平均可能給 3.5 分——看起來還行。但閘門邏輯說：頻率不到 4/5，最高 3 星；再加上其他風險因素，可能只有 2 星。

閘門邏輯的另一個名字是「一票否決」。有些風險是不可補償的。低頻率意味著不確定性，零股限制意味著高門檻——這些不是「扣一點分數」的問題，是「直接出局」的問題。

這個教訓可以推廣到所有評價系統：如果一個維度的低分會造成災難性後果，用閘門邏輯。如果所有維度都是「好一點更好」，用加權平均。

### 教訓五：兩個模式比一個模式更好

歷史模式和年度模式服務兩種完全不同的需求：長期投資者看 CP+，短期買家看星級和倒數計時。強迫合併成一個表格，兩種資訊都會被壓縮和妥協。

兩個模式的代價是：兩組資料抓取、兩組欄位定義、兩組排序邏輯。但共用同一個渲染核心（`table.js`）。差異不在渲染，在資料。這是好的抽象——共同的渲染邏輯、不同的資料模型。

### 教訓六：等候時間要掩飾而不是消除

7.2 秒的載入動畫。Supabase 冷啟動需要 5-7 秒——這個等待時間無法消除。但可以用毛玻璃動畫和進度條掩飾，讓等待看起來不像等待。

另一個例子：15 分鐘 debounce 同步。localStorage 是即時的，使用者感覺不到延遲。Supabase 同步在背景發生，不影響體驗。

原則：使用者感覺到的延遲才是延遲。真實的延遲如果無法消除，就在 UX 層面掩飾。即時回應（localStorage）+ 延遲同步（debounce）比即時同步更好。

## 運營

### 教訓七：免費方案的限制決定了架構

Supabase 免費方案的連線池限制 → 15 分鐘 debounce 同步。
FinMind API 的速率限制 → 200 支輪流更新 + 隨機延遲。
GitHub Pages 的靜態限制 → 所有動態邏輯必須在前端。
GitHub Actions 的超時限制 → 金絲雀檢查避免浪費。

每一個 hack 都是因為免費方案的限制。如果沒有限制，可能會用 WebSocket 即時推送、用 cron job 每分鐘更新、用伺服器端渲染。但有限制的好處是：強制你做最簡方案。

### 教訓八：200 支輪流比分批更新更好維護

另一個方案：把 1,600 支股票分成 8 批，每天更新一批。問題是：哪些股票應該在熱門批？如何處理新股上市？如何處理下市？

200 支「最久沒更新」的方案不需要回答這些問題。它是一個自我調節的佇列——最久沒更新的股票優先更新，冷門股票更新頻率低，熱門股票更新頻率高（因為被點擊多、排名高、需要及時數據）。

### 教訓九：金絲雀檢查節省的不只是 API 呼叫

金絲雀檢查（用台積電測試是不是交易日）節省的不只是 200 次 API 呼叫。它還節省了：

- GitHub Actions 的執行時間（假日不執行）
- Supabase 的寫入次數（假日不更新）
- FinMind 的速率限流額度（假日不浪費）
- 錯誤日誌的噪音（假日不會有大量 null 數據的警告）

一個 if 敘述，四個好處。

## 誠實

### 教訓十：估值為零比虛假估算更可靠

雜物桶的 `base_value: 0`。我知道「面紙」值 20 元，「香皂」值 15 元——但我不知道「功能型隨身盒」值多少。

給一個不准確的估值比給零更危險。因為使用者會信任你的數字。如果「功能型隨身盒」被估算為 30 元，使用者可能會因為這 30 元做出購買決策。但實際上它的價值可能在 10-50 元之間——這個誤差大到失去了參考價值。

估值為零的結果是星級會降到 0 星，使用者在排行榜上看不到這支股票。這才是對的——如果你不知道一個東西值多少，就應該承認不知道，而不是給一個不准確的數字。

### 教訓十一：雜物桶不是偷懶是誠實

雜物桶有 37 個 keywords，包含「油票」、「面紙」、「衛生紙」、「毛巾」、「香皂」等。這些東西的共同特徵是：很難準確估價，且價值通常很低。

把它們歸為「雜物」、估值為零，不是懶得分類。是對不確定性的誠實。如果為每一種雜物都建立一個節點、設定一個估值，給使用者的印象是「我們知道每個紀念品值多少」——但實際上，對於這些長尾項目，我們不知道。

### 教訓十二：匿名 UUID 不完美但夠用

清除 localStorage 會生成新的 UUID。換裝置是新的 UUID。同一個人用兩個瀏覽器會被算成兩個訪客。

但對於零成本的近似統計來說，這些誤差可以接受。我們不需要知道「使用者 A 星期三瀏覽了三支股票然後買了一支」。我們需要知道的是大方向：「哪支股票最多人看」、「每天有多少訪客」、「歷史模式和年度模式哪個更受歡迎」。

**完美的敵人是夠好。** 追求 100% 準確的追踪會導致強制登入，強制登入會趕走 80% 的訪客。80% 的近似數據比 100% 的零數據更有價值。

## 最後一件事

這個專案有一個特點：它是真的有人用的工具，不是展示用作品集。

這意味著每一個技術決策都經過了真實使用場景的驗證。金絲雀檢查不是理論——假日真的會觸發它。15 分鐘 debounce 不是過度設計——真的有人在一分鐘內點擊了 10 支股票。7.2 秒載入動畫不是裝飾——Supabase 冷啟動真的需要這麼長時間。

這些教訓的有效性來自一個事實：它們是在真實使用中發現問題、解決問題的。不是假設問題可能存在就先預防。
